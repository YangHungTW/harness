1.Which command should I run?

START: I want to begin a feature. │ ├─ Need a reviewable plan first, or this feature depends on │ another that's still in-flight? │ │ YES → /yang-toolkit:plan-feature "<desc>" │ ↓ (review .claude/plans/<slug>.md, edit if needed) │ /yang-toolkit:execute-plan │ ↓ (auto-delegates to tdd-feature or feature-dev-tracked) │ └─ NO, just start cycling: │ ├─ Need TDD discipline? → /yang-toolkit:tdd-feature "<desc>" └─ No TDD → /yang-toolkit:feature-dev-tracked "<desc>"
Rule of thumb. Plan-first when the work is large enough that a teammate (or future-you) will want to see the plan in a PR. Direct flow when you already know exactly what to build.

2.Plan-first flow (new in v0.4.0)

Step 1 — draft the plan

# Runs a parallel research fan-out, then writes .claude/plans/<slug>.md:
#   - codebase patterns/conventions to mirror
#   - project history: ledger + prior plans + decision dirs + CLAUDE.md
#   - external recency research (via the deep-research skill) when it helps
# Results become typed Memory References. Research depth is chosen
# automatically (deep for fuzzy/broad/external/risky work, shallow for small
# local changes) -- you do NOT pass a flag.
/yang-toolkit:plan-feature "Add cancellation policy UI to booking flow"

Input can be any captured context, not just a clean description: paste a terminal error, a GitHub issue URL, a meeting transcript, or attach a screenshot / mockup — raw is fine, the probes mine it against your codebase and history. Override the auto depth only if you disagree: --deep forces the full treatment (external research + a "plan-for-the-plan" pass); --shallow forces one light pass.

Then open the produced file and edit:

  • Acceptance Criteria — each item needs Check: (a runnable command in backticks) and Pass: (an observable condition; no fuzzy words like good/nice/clean).
  • Files Touched — paths or globs. Used both as scope guard and as work-split when running in team or workflow mode.
  • Frontmatter — confirm discipline (tdd | normal) and orchestration (single | workflow | team).

Step 2 — dry-run to inspect what it'll do

/yang-toolkit:execute-plan --dry-run

Prints the parsed criteria and resolved deps. In single/team mode it also shows the assembled /goal condition and the downstream command; in workflow mode it shows the args that would be passed to the workflow plus the directory-affinity partition of Files Touched (so you can eyeball whether the slices are actually disjoint). No state is changed.

Step 3 — execute

/yang-toolkit:execute-plan

Runs the plan one of three ways per its orchestration:

  • single (default) — sets /goal from the criteria, switches status: executing, delegates to tdd-feature or feature-dev-tracked by discipline.
  • workflow — deterministic parallel fan-out via the built-in Workflow tool (workflows/execute-plan-team.workflow.js): partitions Files Touched into disjoint slices, implements them concurrently (Phase 1), then verifies every Acceptance Criterion by running its Check command (Phase 2). No /goal loop — the Verify phase is the completion check.
  • team — experimental agent-teams (see callout in §7); kept for back-compat.

On completion (any mode) it writes a ## Execution Log block back into the plan and appends one ledger record with plan_path, orchestration, and mode-specific fields (goal_turns for single/team; workers/criteria_pass/criteria_fail for workflow).

Revising or replanning

CommandEffect
/yang-toolkit:plan-feature --revise <slug>Append a ## Revision N block; preserve original; refresh <!--auto--> Memory References, keep user-added ones.
/yang-toolkit:plan-feature --from <slug>Replan from scratch (overwrites the file after confirmation).

Sharing a plan with a non-terminal reviewer

A plan.md is great in a terminal but awkward to hand to someone in Slack/a browser. Render it to a clean, self-contained HTML document:

# Writes .claude/plans/<slug>-<TS>.html -- a read-only, timestamped snapshot
# (frontmatter -> header, acceptance criteria -> checkboxes). Safe to send/airdrop.
/yang-toolkit:share-plan <slug>

It never touches the .md (the source of truth). Revise via plan-feature --revise, then re-run share-plan for a fresh snapshot.

Execute overrides

FlagUse when
--from <slug>Run an explicit plan instead of the most-recent draft.
--single / --workflow / --teamOverride the plan's orchestration for this run only.
--autoEnter auto mode for this run so the /goal loop (single/team) or the workflow's agents run unattended. Without it each turn still pauses for tool approval -- /goal alone only removes per-turn prompts.
--no-goalSkip /goal setup; you control turn-by-turn.
--ignore-depsProceed even if depends_on items aren't done (recorded in ledger).
--dry-runParse + show /goal + delegate target (single/team) or workflow args + partition map (workflow); no execution.
Fuzzy-word lint. /execute-plan rejects any Pass: line containing good, acceptable, nice, clean, mostly, properly, correctly, robust, performant, scalable, maintainable, sufficient, reasonable. Override the whole list with .claude/plans/.fuzzy-words (one entry per line; replaces, doesn't merge).

3.Direct flow (no plan stage)

Same as before v0.4.0 — unchanged. Use these when the feature is small enough that a plan artifact is overhead.

# Regular
/yang-toolkit:feature-dev-tracked "Add cancellation policy UI"

# TDD discipline
/yang-toolkit:tdd-feature "Add cancellation policy UI"

# Pick up a paused feature-dev-tracked session and continue with TDD
/yang-toolkit:tdd-feature   # reads .claude/state/current-feature.txt

Both write per-phase decision docs under docs/decisions/{date}-{slug}/ and append one ledger record on completion.

# From commit-commands plugin

/yang-toolkit:ledger-append --close <slug>

# Manual backfill (interactive) / preview before writing

6.Passive (no commands; just runs)

Session ends (Stop)Play a short completion sound so you can tell which of several parallel sessions just finished. Opt out with HARNESS_DISABLE_NOTIFY=1; override the sound with HARNESS_NOTIFY_SOUND=/path; add a desktop notification with HARNESS_NOTIFY_DESKTOP=1.

7.Claude Code built-ins worth pairing

These aren't part of the toolkit but the new plan flow leans on them.

# Plan mode
claude --permission-mode plan        # on launch
Shift+Tab                            # toggle mid-session

# /goal — auto-loop until a condition holds (execute-plan uses this internally)
/goal <condition>
/goal                                # show status
/goal clear                          # cancel

# Parallel sessions
claude --worktree feature-auth
claude --continue                   # resume most-recent session
claude --resume                     # pick from list
Workflow mode (orchestration: workflow) is the recommended parallel option. It fans out via the built-in Workflow tool, partitioning Files Touched into disjoint slices that build concurrently, then verifying each Acceptance Criterion. No experimental flag needed. Precondition: the slices must be separable by directory — if workers would all need the same shared file, prefer --single (the command warns and surfaces any scopeViolations if they overlap).
Agent teams (used when a plan has orchestration: team) require CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 in env or settings. Without it, execute-plan warns and falls back to single. Prefer workflow unless you specifically want long-lived teammates that message each other.

9.Mental model

plan = WHY / WHAT, the part a reviewer reads.
decision dir = HOW, the per-phase process record.
ledger = cross-feature statistics.
CLAUDE.md = rules that have crystallised out of past work.

/plan-feature writes the first. /execute-plan connects all four — it produces the decision dir (via delegation), appends to the ledger on close-out, and triggers curate-claude-md so learnings flow back into the rules.
