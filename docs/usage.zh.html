1.該下哪個指令？

起點：我要開始一個 feature。 │ ├─ 需要先寫一份可以被 review 的計畫？ │ 或這個 feature 依賴另一個還沒做完的 feature？ │ │ YES → /yang-toolkit:plan-feature "<描述>" │ ↓ (打開 .claude/plans/<slug>.md 檢視、視需要編輯) │ /yang-toolkit:execute-plan │ ↓ (自動委派給 tdd-feature 或 feature-dev-tracked) │ └─ NO，直接開始 cycling： │ ├─ 需要 TDD 紀律？ → /yang-toolkit:tdd-feature "<描述>" └─ 不用 → /yang-toolkit:feature-dev-tracked "<描述>"
判斷原則。 工作大到隊友（或未來的自己）會想在 PR 裡看到計畫 → 走計畫先行。 已經很清楚要做什麼 → 直接動手。

2.計畫先行流程（v0.4.0 新增）

Step 1 — 起草計畫

# 先做平行 research fan-out，再寫出 .claude/plans/<slug>.md：
#   - 要照抄的 codebase 慣例／pattern
#   - 專案歷史：ledger + 既有 plans + decision dirs + CLAUDE.md
#   - 外部最新研究（透過 deep-research skill），有需要才跑
# 結果變成 typed Memory References。研究深度「自動判斷」
# （模糊／大範圍／外部／高風險 → 深；小範圍本地改動 → 淺），你不用下 flag。
/yang-toolkit:plan-feature "Add cancellation policy UI to booking flow"

輸入可以是任何擷取到的素材，不限乾淨描述：貼終端機錯誤、 GitHub issue URL、會議逐字稿，或直接附上截圖／設計稿 — raw 就好，probe 會拿它 對你的 codebase 與歷史做萃取。只有在你不同意自動深度時才覆寫： --deep 強制完整流程（外部研究 + 「先規劃怎麼規劃」那一步）； --shallow 強制單輪輕量。

打開檔案編輯：

  • Acceptance Criteria（驗收條件）— 每條都要有 Check:（包在 backtick 裡的可執行命令）跟 Pass:（可觀測條件；不能用模糊詞如 good、nice、clean）。
  • Files Touched（涉及檔案）— 路徑或 glob。同時作為範圍守衛、以及 team / workflow mode 下的分工依據。
  • frontmatter — 確認 discipline（tdd | normal）和 orchestration（single | workflow | team）。

Step 2 — dry-run 看它會做什麼

/yang-toolkit:execute-plan --dry-run

印出解析後的 criteria 和已解析的相依。single/team 模式還會印組好的 /goal condition 跟下游指令；workflow 模式則印要傳給 workflow 的 args，外加 Files Touched 依目錄親緣性的分割結果（讓你先目視確認各 slice 是否真的不相交）。不改任何狀態。

Step 3 — 真的執行

/yang-toolkit:execute-plan

依 plan 的 orchestration 用三種方式之一執行：

  • single（預設）— 從 criteria 設好 /goal、把狀態改成 executing、依 discipline 委派到 tdd-feature 或 feature-dev-tracked。
  • workflow — 透過內建 Workflow 工具做確定性平行 fan-out（workflows/execute-plan-team.workflow.js）：把 Files Touched 切成不相交的 slice 平行實作（Phase 1），再逐條跑每個 Acceptance Criterion 的 Check 指令驗收（Phase 2）。不用 /goal 迴圈 — Verify 階段就是完成判定。
  • team — 實驗性 agent-teams（見 §7 callout）；保留向後相容。

任一模式完成時，都會把 ## Execution Log 區塊寫回 plan，並 append 一筆 ledger record（含 plan_path、orchestration，以及各模式專屬欄位：single/team 是 goal_turns；workflow 是 workers/criteria_pass/criteria_fail）。

修訂或重新規劃

指令效果
/yang-toolkit:plan-feature --revise <slug>追加 ## Revision N 區塊；保留原文；refresh <!--auto--> 的 Memory References，保留 user 加的 bullet。
/yang-toolkit:plan-feature --from <slug>從頭重寫（確認後覆蓋整份檔案）。

把計畫分享給不用終端機的人

plan.md 在終端機裡很好用，但丟給在 Slack／瀏覽器的人很尷尬。 把它渲成一份乾淨、自包含的 HTML 文件：

# 寫出 .claude/plans/<slug>-<TS>.html — 唯讀、有時間戳的快照
# （frontmatter → 標頭，acceptance criteria → checkbox）。可直接寄出／AirDrop。
/yang-toolkit:share-plan <slug>

它不會動到 .md（真實來源）。要改就用 plan-feature --revise，再重跑 share-plan 產生新快照。

Execute 的 override flag

Flag使用時機
--from <slug>指定執行某個 plan，不要拿最新的 draft。
--single / --workflow / --team本次蓋掉 plan 內的 orchestration。
--auto本次進入 auto mode，讓 /goal 迴圈（single/team）或 workflow 的 agent 無人值守。沒加的話每輪還是會停下等 tool approval — /goal 只負責省掉 per-turn 確認。
--yes非互動：跳過「確認後繼續」類的提示（Step 3 的 /goal 條件確認、單一 match 的確認）。真正的決策走安全預設 — executing 自動 resume、failed 或 --auto 開不起來則 abort。搭配 --auto 就是完全無人值守；loop --unattended 內部就用這組。
--no-goal不設 /goal，由你自己 step through。
--ignore-deps強制執行，即使 depends_on 還沒 done（會記在 ledger）。
--dry-run解析 + 顯示 /goal + 委派目標（single/team）或 workflow args + 分割預覽（workflow）；不執行。
模糊詞檢查。 /execute-plan 會拒絕 Pass: 行含有： good、acceptable、nice、 clean、mostly、properly、 correctly、robust、performant、 scalable、maintainable、sufficient、 reasonable。要客製化整個 list 就建立 .claude/plans/.fuzzy-words（一行一個，**取代**預設不合併）。

3.直接動手流程（沒有 plan 階段）

v0.4.0 之前的用法 — 沒變。Feature 小到不需要 plan artifact 時用這個。

# 一般
/yang-toolkit:feature-dev-tracked "Add cancellation policy UI"

# TDD 紀律
/yang-toolkit:tdd-feature "Add cancellation policy UI"

# 接續一個暫停的 feature-dev-tracked，轉入 TDD
/yang-toolkit:tdd-feature   # 讀 .claude/state/current-feature.txt

兩條都會在 docs/decisions/{date}-{slug}/ 寫各 phase 的決策文件，並在完成時 append 一筆 ledger record。

5.隨時可用（讀資料）

指令用途
/yang-toolkit:status一頁總覽：進行中的 feature、各 plan 狀態、ledger 尾巴，外加一條下一步建議。--abandon 一步收掉進行中的 feature（ledger + state + plan 三處一次處理）。
/yang-toolkit:loopplan/ledger 感知的 in-session 心跳：每個 tick 找出下一個可執行的 plan（accepted/draft），交給 execute-plan 在客觀驗收條件閘後執行，記錄結果，再排下一個 ScheduleWakeup tick。預設 propose-only + 單次；--unattended 才進入自動執行 + 循環心跳（只跑 accepted，並以 permissions.deny 為硬底線）— 它用 execute-plan --auto --yes 執行，所以 tick 不會卡在確認提示上；--max-tokens <N> 設總花費上限（Ralph-Wiggum 防呆）。停止：Esc 或 CLAUDE_CODE_DISABLE_CRON=1。是內建 /loop skill 的 plan 感知版。
/yang-toolkit:today當日摘要：聚合 GitHub / Jira / Slack 等外部 surface + 所有追蹤 repo 的最近 ledger entries。
/yang-toolkit:week跨 repo 週報，來源是 ~/.config/harness/repos.json。
/yang-toolkit:dashboard把 .claude/ledger.jsonl + 即時 git 渲成一組有時間戳的成對檔：互動式 dashboard-<TS>.html（timeline + kanban + stats + loop economics〔接受率 & 每個被接受變更的成本〕，含 outcome 篩選、feature focus、與瀏覽器內的 git diff 檢視）以及給 review/AI 用的純文字 dashboard-<TS>.md。
/yang-toolkit:share-plan <slug>把計畫（或任何 plan 形狀的 markdown）渲成乾淨、自包含、可分享的 HTML，輸出在 .claude/plans/<slug>-<TS>.html。唯讀快照。
/yang-toolkit:claude-md-gaps檢視被動 hook 累積的 nested CLAUDE.md 候選。要 user accept 才會真的寫檔。
/yang-toolkit:curate-claude-md稽核並重整既有 CLAUDE.md（技術規則往上漂、業務規則往下沉）。

6.被動觸發（不用下指令）

觸發點動作
每次 tool call（PreToolUse）append 到 .claude/logs/session-YYYYMMDD.jsonl。
Edit / Write / MultiEdit（PostToolUse）評分當前 folder 對 CLAUDE.md 的需要 → .claude/state/claude-md-candidates.jsonl。
改了 production code 沒動測試下一輪 context 注入一次性提醒（per-session 去重）。設 HARNESS_DISABLE_TEST_PARITY=1 可關。
Subagent 結束（SubagentStop）把當前 agent 名稱寫到 .claude/state/current-agent.txt（餵 statusline + dashboard）。
Session 結束（Stop）append 摘要到 .claude/ledger.jsonl，outcome: in-progress。
Session 結束（Stop）播一聲完成提示音，平行多 session 時知道哪個跑完了。HARNESS_DISABLE_NOTIFY=1 關閉；HARNESS_NOTIFY_SOUND=/path 換音效；HARNESS_NOTIFY_DESKTOP=1 加桌面通知。

7.值得搭配的 Claude Code 內建

不是 toolkit 的一部分，但新的 plan flow 會用到。

# Plan mode
claude --permission-mode plan        # 啟動時
Shift+Tab                            # session 中切換

# /goal — 自動 loop 到條件成立（execute-plan 內部就用這個）
/goal <condition>
/goal                                # 看狀態
/goal clear                          # 取消

# 平行 session
claude --worktree feature-auth
claude --continue                   # 接最近一個 session
claude --resume                     # 從清單選
Workflow 模式（orchestration: workflow）是建議的平行選項。 它透過內建 Workflow 工具 fan-out，把 Files Touched 切成不相交的 slice 平行建置，再逐條驗收 Acceptance Criterion，不需任何實驗性 flag。前提：各 slice 要能依目錄分開 — 若所有 worker 都得改同一個共用檔，請改用 --single（重疊時 指令會警告並把 scopeViolations 浮出來）。
Agent teams（當 plan 設 orchestration: team 時用） 需要 env 或 settings 裡有 CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1。 沒設的話，execute-plan 會警告並 fallback 到 single。 除非你特別想要會互相傳訊的長壽 teammate，否則優先用 workflow。

8.每個 repo 的檔案配置

client-repo/
├── .claude/
│   ├── ledger.jsonl       # COMMIT — 跨 feature 的專案記憶
│   ├── plans/             # COMMIT .md — 可 review 的計畫 artifact
│   │   ├── <slug>.md      # 每個被規劃的 feature 一份；status 追蹤生命週期
│   │   ├── <slug>-<TS>.html # gitignore — /share-plan 匯出（可重新產生）
│   │   └── .fuzzy-words   # OPTIONAL — 覆寫預設的模糊詞 lint
│   ├── logs/              # gitignore — 吵雜的 per-session tool calls
│   ├── state/             # gitignore — 短暫狀態
│   │   ├── current-agent.txt
│   │   ├── current-feature.txt            # feature-dev-tracked / execute-plan 設
│   │   ├── claude-md-candidates.jsonl     # 待處理的 CLAUDE.md 提案
│   │   └── test-parity-warned-YYYYMMDD.txt
│   ├── dashboard-<TS>.html # gitignore — 互動式 render（有時間戳）
│   └── dashboard-<TS>.md   # gitignore — 純文字 review/AI render（以 TS 配對）
└── docs/decisions/        # COMMIT — 每個 feature 的決策軌跡
    └── 2026-05-28-<slug>/
        ├── 01-discovery.md
        ├── 02-architecture.md
        ├── 02b-test-plan.md      # 只 TDD 才有
        ├── 03-implementation.md  # 或 03-tdd-cycles.md
        ├── 04-review.md
        └── 05-summary.md         # 終點 — 寫完之後拒絕再進入

9.心智模型

plan = WHY / WHAT，給 review 的人看的部分。
decision dir = HOW，per-phase 的過程記錄。
ledger = 跨 feature 的統計。
CLAUDE.md = 從過往工作沉澱出來的規則。

/plan-feature 寫第一個。 /execute-plan 把四者串起來 — 透過委派產出 decision dir、 收尾時 append ledger、並觸發 curate-claude-md 讓 learnings 回流到規則。
